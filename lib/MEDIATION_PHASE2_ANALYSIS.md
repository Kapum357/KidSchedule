## 1. Message Repository (`lib/persistence/postgres/messaging-repository.ts`)

### Key Finding: Hash Chain Architecture
Messages use a **cryptographically-verified hash chain** for tamper-evident communications. This is important for legal/evidentiary purposes in co-parenting disputes.

### Creating Messages

**Method Signature:**
```typescript
async create(data: Omit<DbMessage, "id" | "createdAt" | "updatedAt">): Promise<DbMessage>
```

**Required Parameters (`DbMessage`):**
```typescript
interface DbMessage {
  id: string;                           // Generated by database
  threadId: string;                     // REQUIRED: Which thread this belongs to
  familyId: string;                     // REQUIRED: Family context
  senderId: string;                     // REQUIRED: Parent ID of sender
  body: string;                         // REQUIRED: Message content
  sentAt: string;                       // REQUIRED: ISO-8601 timestamp
  readAt?: string;                      // Optional: When recipient read it
  attachmentIds: string[];              // Optional: File references from Vault
  toneAnalysis?: {                      // Optional: Claude analysis result
    isHostile: boolean;
    indicators?: string[];
  };
  messageHash: string;                  // Auto-computed by repository
  previousHash?: string;                // Auto-linked to previous message
  chainIndex: number;                   // Auto-incremented per thread
  createdAt: string;                    // Auto-set by database
  updatedAt: string;                    // Auto-set by database
}
```

### Important Behaviors
1. **Hash Chain Integrity**: The repository automatically:
   - Fetches the last message in the thread
   - Computes `chainIndex` (last index + 1)
   - Links `previousHash` to the previous message's hash
   - Computes a new `messageHash` using `computeMessageHash()`

2. **Thread Update**: Calling `create()` automatically updates the thread's `last_message_at` timestamp

3. **Immutability**: Messages cannot be updated after creation (hash chain requirement). There is no `update()` method; it throws an error.

### Integration Points
- **Server Actions**: Use `db.messages.create()` to persist suggestions sent from mediation page
- **Tone Analysis**: The `toneAnalysis` field stores Claude's assessment; populate this when storing AI-generated suggestions
- **Timestamps**: Always use ISO-8601 format for `sentAt`

## 3. Message API Patterns

### Authentication Pattern
All message endpoints (including mediation message creation) follow this authentication sequence:

```typescript
// 1. Authenticate user
const user = await getAuthenticatedUser();
if (!user) {
  return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
}

// 2. Get database instance
const db = getDb();

// 3. Fetch parent record (linked to userId)
const parent = await db.parents.findByUserId(user.userId);
if (!parent) {
  return NextResponse.json({ error: "Parent profile not found" }, { status: 404 });
}

// 4. Fetch family (ensures parent owns this family)
const family = await db.families.findByParentUserId(user.userId);
if (!family) {
  return NextResponse.json({ error: "Family not found" }, { status: 404 });
}
```

### Response Format Pattern
```typescript
return NextResponse.json(
  {
    id: resource.id,
    status: "success",
    // ... other fields
  },
  { status: 201 } // or 200, 400, 404, 409, etc.
);
```

### Error Responses
- `400`: Bad request (validation failure)
- `401`: Unauthorized (not authenticated)
- `403`: Forbidden (authenticated but not authorized for this family)
- `404`: Not found (resource doesn't exist)
- `409`: Conflict (e.g., duplicate enrollment)

### Note on References
This pattern is documented in `app/api/messages/relay/route.ts` (SMS relay enrollment context), but the same auth, validation, and response conventions apply across all message endpoints. The key is to authenticate first, validate inputs, authorize family access, then call the repository layer.

## 4. Claude Integration (`lib/providers/ai/mediation-assistant.ts`)

### Two Functions Available

#### A. `getMediationAssistantTips()`
Analyzes recent messages in a conversation and returns de-escalation insights to help parents communicate better.

**Function Signature:**
```typescript
export async function getMediationAssistantTips(
  userId: string,
  recentMessages: Message[]
): Promise<MediationAssistantResult>
```

**Input:**
- `userId`: Parent's user ID (for rate limiting & logging)
- `recentMessages`: Array of `Message` objects (up to 15 most recent used)

**Output:**
```typescript
interface MediationAssistantResult {
  conflictLevel: "low" | "medium" | "high";
  deescalationTips: string[];  // 3-5 practical tips
}
```

**When to Use:**
- Use to generate de-escalation suggestions FROM recent messages in a thread
- Called when a parent wants AI-powered tips on how to improve future replies
- Input: historical messages; output: proactive advice

**Key Details:**
- Automatically redacts PII before sending to Claude (emails, phones, SSNs, card numbers)
- Uses `claude-3-5-haiku-latest` by default (configurable via `CLAUDE_MEDIATION_MODEL` env var)
- Falls back to local `MediationSuggestionEngine` if Claude is unavailable
- Rate limited to 100 requests per user per minute
- Circuit breaker opens if error rate exceeds 50% over 5 minutes

#### B. `analyzeMessageTone()`
Analyzes a single message (typically a draft) for hostile language and provides a neutral rewrite.

**Function Signature:**
```typescript
export async function analyzeMessageTone(
  userId: string,
  messageText: string
): Promise<ToneAnalysisResult>
```

**Output:**
```typescript
interface ToneAnalysisResult {
  isHostile: boolean;           // Flag for hostile content
  indicators: string[];         // List of hostility markers (e.g., "ALL_CAPS", "Threats")
  neutralRewrite: string;       // Claude's suggested rewrite
}
```

**When to Use:**
- Use to CHECK a single message before sending (e.g., parent draft, AI suggestion)
- Called to validate tone and get a neutral rewrite if the message is hostile
- Input: a single message; output: tone flags + suggested revision

### Claude Integration Architecture (`lib/providers/ai/claude-adapter.ts`)

#### Core Function: `runClaudeJsonWithGuardrails<T>()`
All Claude calls use this function, which provides:

**Safeguards:**
- **User rate limiting**: 100 calls/minute per user
- **Circuit breaker**: Opens if >50% error rate over 5 minutes (60-second backoff)
- **PII redaction**: Before Claude sees any text
- **Graceful fallback**: Returns a default value if Claude unavailable
- **Cost tracking**: Logs estimated USD cost per call
- **Observability**: Tracks token usage, latency, and errors

**Input Type:**
```typescript
type RunClaudeJsonInput<T> = {
  userId: string;
  operation: "tone_analysis" | "mediation_assistant";
  model: string;  // e.g., "claude-3-5-haiku-latest"
  maxTokens: number;
  systemPrompt: string;
  userPrompt: string;
  fallback: T;   // What to return if Claude unavailable
  validate: (value: unknown) => T;  // Parsing/validation function
};
```

**Usage Pattern:**
```typescript
return await runClaudeJsonWithGuardrails<MyResultType>({
  userId: "user-123",
  operation: "tone_analysis",
  model: "claude-3-5-haiku-latest",
  maxTokens: 280,
  fallback: DEFAULT_RESULT,
  systemPrompt: "...",
  userPrompt: "...",
  validate: parseMyResult,  // Function that validates JSON from Claude
});
```

#### Helper: `redactPIIForClaude()`
Before sending any text to Claude, call this function:

```typescript
const redactedText = redactPIIForClaude(messageBody);
// Redacts:
// - Emails: name@example.com → [redacted_email]
// - Phones: +1 555-1234 → [redacted_phone]
// - SSNs: 123-45-6789 → [redacted_ssn]
// - Credit cards → [redacted_card]
```

## 6. Integration Workflow for Phase 2

### Step 1: Send Mediation Suggestion (Server Action)
1. Get the mediation suggestion text (AI-generated or admin-crafted)
2. Call `getMediationAssistantTips()` to get the conflict level
3. Use `analyzeMessageTone()` to check tone
4. Create a message thread or find existing one
5. Call `db.messages.create()` with:
   - `threadId`: Thread where suggestion is posted
   - `familyId`: Family context
   - `senderId`: Use the requesting parent's user ID (they initiated the suggestion, even if AI-assisted). Do NOT use a system actor ID—this ensures audit trails accurately reflect which parent authored the suggestion.
   - `body`: Suggestion text
   - `sentAt`: Current ISO timestamp
   - `toneAnalysis`: Result from `analyzeMessageTone()`

### Step 2: Adjust Tone via Claude (Server Action)
**IMPORTANT: Messages are immutable after creation (hash chain requirement).**

1. Take the current suggestion text (before creation)
2. Call `analyzeMessageTone()` to get the tone analysis
3. If `neutralRewrite` is provided and you want to use a better version:
   - **Option A (Recommended)**: Use Claude's `neutralRewrite` as the message body BEFORE calling `db.messages.create()`. This ensures the final message in the thread has the improved tone from the start.
   - **Option B**: If adjustment happens after creation, create a NEW message in the same thread with the adjusted text. Link it to the original via `attachmentIds` or thread context (e.g., "Revised version of previous suggestion").
4. Never attempt to update an existing message—the repository will reject it due to hash chain integrity.

### Step 3: Warning Dismissal with Message
1. Fetch the mediation warning
2. Create a message in the same thread
3. Call `db.messages.create()` to record the dismissal action
4. Link the warning ID in `attachmentIds` or use a metadata field
